# File-upload safety

## Introduction

This document explains how to accept a file from an untrusted user without
letting that file become an attack. Accepting an upload means accepting bytes
that a stranger chose, along with metadata the stranger also chose — a claimed
file type and a filename — and none of that attacker-controlled metadata can be
trusted. The safe path treats every upload as hostile until the server itself
has inspected the actual bytes, bounded how many of them it will accept, and
stored the result under a name the server generated rather than one the user
supplied.

This document uses two terms throughout. A Multipurpose Internet Mail
Extensions (MIME) type is the short label, like `application/pdf`, that names
what kind of content a chunk of bytes is. Magic-byte validation is the
technique of reading the first few bytes of a file and comparing them against
the known signatures of real file formats to decide that type from the content
itself, instead of believing a label sent alongside it.

**Learning outcomes** — after reading this document you will be able to:

- Explain why the client-declared content type and filename are untrustworthy and why the server must detect the true type from the file's own leading bytes.
- Describe the four core upload-safety controls: content-based type detection, a hard size limit, server-generated object keys, and retaining the original filename as display-only data.
- Recognize the symptoms of common upload-handling mistakes and apply the recovery for each.

## Problem it solves

A server that accepts uploads faces an attacker who controls every byte and
every piece of metadata in the request. The concrete problems are: a file that
claims to be a harmless document but is actually an executable or a script; a
file so large it exhausts the server's memory or disk; a small file that
expands to an enormous size when opened; and a filename crafted to escape the
intended directory and overwrite a system file. Each of these turns a feature —
"let users upload a resume" — into a way to crash, exploit, or take over the
service.

The common prior approach is to trust the request. A naive handler reads the
`Content-Type` header the client sent, believes it, saves the file under the
name the client provided, and imposes no size ceiling. That approach fails on
every front:

- The declared content type is a free-text field the client fills in. An attacker sets it to `application/pdf` on a file that is not a document at all, and the trusting server stores and later processes it as one.
- The supplied filename can contain path segments such as `../../etc/passwd`. Used directly as a storage path, it lets the upload land somewhere it was never meant to go.
- With no size limit, one request carrying a multi-gigabyte body, or a tiny compressed file that inflates a thousand-fold when decompressed, is enough to take the server down.

Content-based validation, hard limits, and server-generated names close those
holes by refusing to let attacker-controlled metadata steer any decision.

## Mental model

Think of a parcel arriving at a secure mailroom. The label on the outside is
written by the sender and cannot be trusted, so the mailroom follows a fixed
procedure:

1. A parcel arrives with a label claiming "Books" (the declared content type) and a return address scrawled by the sender (the filename). The clerk ignores both for any safety decision.
2. The clerk weighs the parcel first. If it is over the posted weight limit, it is rejected at the door before anyone opens it (the hard size limit).
3. The clerk opens the parcel and inspects what is actually inside, comparing the real contents against the short list of items the mailroom accepts (magic-byte type detection). A box of "Books" that actually contains something else is refused.
4. An accepted parcel is given a fresh internal tracking number generated by the mailroom (the server-generated key). It is filed under that number, never under the sender's scrawled return address.
5. The sender's original label is photographed and kept in a logbook for reference only (the display-only original filename). It is never used to decide where the parcel is stored.

The lesson of the mailroom is that the only trustworthy facts are the ones the
mailroom establishes itself — the weight it measured and the contents it
inspected. Everything the sender wrote is treated as a claim, not a fact.

## How it works

Every upload arrives as part of a request that carries three separable things:
the raw bytes of the file, a declared content type, and a declared filename.
The declared content type and filename are supplied by the sender and are
therefore claims, not evidence. Safe handling rests on the principle that a
safety decision is only made from something the server can verify for itself.

The first control is content-based type detection. Real file formats begin
with a recognizable signature — a fixed sequence of leading bytes sometimes
called a magic number. A Portable Document Format (PDF) file starts with the
characters `%PDF`; a modern word-processing document is a compressed archive
with a recognizable internal structure. A detector reads those leading bytes
and reports the true type, independent of any label the sender attached. If the
detected type is not on the short allowlist of accepted formats, the upload is
refused. This defeats a spoofed content-type header, because the header is
never consulted for the decision.

The second control is a hard size limit enforced before the bytes are stored or
processed. The server checks the declared length of the body and refuses
anything over a fixed ceiling, returning an error rather than reading the rest.
A separate guard protects against a decompression attack, where a small
compressed file is crafted to expand into an enormous one: before opening such
an archive, the handler sums the declared uncompressed sizes of its entries and
refuses the file if that total crosses a ceiling. Both limits convert an
unbounded resource demand into a bounded, predictable one.

The third control concerns naming. The sender's filename is never used to build
a storage location, because a crafted name can contain path-traversal segments
that escape the intended directory. Instead the server generates its own unique
identifier for each accepted file and stores the bytes under that identifier. A
Universally Unique Identifier (UUID) — a 128-bit value with an astronomically
small chance of collision — is a natural choice, since a fresh one can be
minted for every upload without coordination.

The fourth control is what to do with the original filename. It still has value:
the user expects to see the name they uploaded. So it is kept, but only as
display data — a field shown back to its owner and never used to derive a path,
a key, or any filesystem location. Separating "the name we store under" from
"the name we show" keeps the useful part of the filename while denying it any
power over where bytes land.

## MatchLayer Phase 1 usage

MatchLayer accepts resume uploads at the `POST /api/v1/resumes` endpoint, whose
handler lives at `apps/api/src/matchlayer_api/api/resumes/router.py`. The
handler delegates the actual safety work to two small, single-purpose modules:
the magic-byte detector at `apps/api/src/matchlayer_api/core/mime.py` and the
object-key minter at `apps/api/src/matchlayer_api/core/storage.py`. The
orchestration that ties them together is in
`apps/api/src/matchlayer_api/services/resumes.py`.

The hard size ceiling is checked in the router before any bytes are read or any
object is written. The declared length is compared against a configured maximum
and an oversized request is refused immediately:

Source: `apps/api/src/matchlayer_api/api/resumes/router.py`

```python
    declared_length = _declared_upload_length(upload=file, request=request)
    if declared_length is not None and declared_length > settings.resume_max_bytes:
        raise PayloadTooLargeError(
            f"Uploaded file exceeds the maximum size of {settings.resume_max_bytes} bytes."
        )
```

That ceiling is a single configured value (five mebibytes in Phase 1), declared
once in the settings so it is enforced consistently:

Source: `apps/api/src/matchlayer_api/config.py`

```python
    resume_max_bytes: int = 5_242_880
```

Type detection ignores the client-declared content type entirely. The detector
reads the file's leading bytes, asks a pure-Python format library what they
really are, and returns one of the two accepted kinds or nothing at all — and
"nothing" is what the upload service turns into a rejection:

Source: `apps/api/src/matchlayer_api/core/mime.py`

```python
def detect(data: bytes) -> Literal["pdf", "docx"] | None:
    match = filetype.guess(data)
    if match is None:
        return None
    extension = str(match.extension)
    if extension not in _ACCEPTED_EXTENSIONS:
        return None
    if extension == "pdf":
        return "pdf"
    return "docx"
```

Once a file is accepted, it is stored under a freshly generated, filename-free
key. The key's stem is a time-ordered UUID and its extension comes from the
detected kind — never from the user's filename:

Source: `apps/api/src/matchlayer_api/core/storage.py`

```python
def build_object_key(extension: ResumeKind) -> str:
    return f"{uuid7()}.{extension}"
```

The user's original filename is still kept, but as a display-only column on the
resume record. The schema field that carries it documents exactly that
constraint — it is returned to its owner for display and never used to derive
the storage key or any path:

Source: `apps/api/src/matchlayer_api/api/resumes/schemas.py`

```python

```

Together these realize the four controls end to end: the router bounds the
size, the detector establishes the true type from the bytes, the storage layer
names the object with a server-minted identifier, and the original filename
survives only as something to show the user.

## Common pitfalls

- **Mistake:** Trusting the request's `Content-Type` header (or the filename extension) to decide whether an upload is an accepted type.
  **Symptom:** A file that is not a real document — an executable, a script, or a plain archive renamed with a `.pdf` extension — is accepted and stored, and downstream code that assumes a valid document fails or is exploited when it opens the bytes.
  **Recovery:** Detect the true type from the file's leading bytes server-side and compare the detected type against an allowlist; never read the declared content type or extension for the decision.

- **Mistake:** Reading the entire upload into memory before checking its size, or imposing no size ceiling at all.
  **Symptom:** A single large request, or a burst of them, spikes memory usage and the server slows to a crawl or is killed by the host for exceeding its memory limit.
  **Recovery:** Enforce a hard maximum size against the declared length before reading the body, and reject anything over the ceiling with a "payload too large" error.

- **Mistake:** Using the user-supplied filename as the storage key or as part of a filesystem path.
  **Symptom:** Two users uploading `resume.pdf` overwrite each other, or a crafted name containing `../` segments writes outside the intended location and clobbers another file.
  **Recovery:** Generate a unique server-side identifier for every accepted file and store under that; keep the original filename only in a separate display-only field.

- **Mistake:** Opening a compressed upload (such as the archive inside a word-processing document) without bounding how much it can expand.
  **Symptom:** A small uploaded file consumes gigabytes of memory or disk when decompressed, exhausting the server from a request that looked tiny on the wire.
  **Recovery:** Before decompressing, sum the declared uncompressed sizes of the archive's entries and refuse the file if the total or the entry count crosses a fixed ceiling.
